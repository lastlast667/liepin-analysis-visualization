# Data Processing Rules - 拉勾网招聘数据处理规则

> 本文件定义拉勾网招聘数据从原始爬取到入库的全链路处理规则。

---

## 一、数据标注规则（核心）

### 1.1 核心原则

**标注依据：搜索关键词默认标注 + 标题校验修正。**

- **搜索关键词**决定**默认标签**（如搜"Java"→默认"Java后端开发"）
- **标题校验**决定是否**修正标签**（如搜"Java"但标题是"Python开发"→修正为"Python/Go后端开发"）
- **岗位标签列表（labels）不参与标注**，仅作为特征用于后续ML向量化

### 1.2 为什么搜索关键词可以作为默认标注

kd 关键词搜索匹配的是标题和描述中的关键词，结果与关键词高度相关：
- 搜"Java" → 结果几乎都是 Java 相关岗位
- 搜"爬虫工程师" → 结果几乎都是爬虫岗位
- 搜"嵌入式" → 结果几乎都是嵌入式岗位

这比 tagCodeList 筛选更准确，因为 tagCodeList 是平台预设的分类，可能漏掉小众岗位。

### 1.3 标注流程

```
全部原始数据 → URL去重 → 统一标注 → 清洗 → 预处理
```

### 1.4 标注规则

**第一步：搜索关键词默认标注**

| 搜索关键词 | 默认 train_label |
|-----------|-----------------|
| Java | Java后端开发 |
| Python | Python/Go后端开发 |
| Go | Python/Go后端开发 |
| C++ | C++后端开发 |
| PHP | 其他后端开发 |
| 爬虫 | Python/Go后端开发 |
| 嵌入式 | C++后端开发 |
| 前端 | 前端开发 |
| 数据分析 | 数据分析 |
| 算法 | 算法工程师 |
| 测试 | 测试/运维开发 |
| 运维 | 测试/运维开发 |
| 全栈 | 其他后端开发 |

**第二步：标题冲突校验**

如果岗位标题与搜索关键词**严重不符**，以标题为准修正标签。

**冲突判定（需修正）：**
- 搜索关键词=Java，标题含 "Python" 且不含 "Java" → 修正为 Python/Go后端开发
- 搜索关键词=Python，标题含 "Java" 且不含 "Python" → 修正为 Java后端开发
- 搜索关键词=前端，标题含 "Java后端" → 修正为 Java后端开发
- 搜索关键词=算法，标题含 "测试工程师" → 修正为 测试/运维开发

**不冲突（保留默认标签）：**
- 标题同时含 "Java" 和 "Go"（正常多技能要求）
- 标题模糊如 "后端开发工程师"（无法明确判断，保留默认）
- 标题含搜索关键词的同义词（如搜"Java"，标题含"JVM"）

**第三步：保留默认**

标题与搜索关键词没有明显冲突 → 保留搜索关键词决定的默认标签。

### 1.5 为什么 labels 不参与标注

- 标签由招聘方填写，可能存在疏忽（如 Java 岗标签未选 Java）
- 标题是招聘方最关注的表达，优先级最高
- 搜索关键词已经提供了强烈的类别信号
- labels 仅作为特征用于后续 ML 向量化

### 1.6 单池搜索设计

本系统采用**单池搜索**：模型预测主类别后，只在该类别的 JD 池内做相似度匹配。

**不做多池搜索，不存储 sub_labels。**

**原因：**
- 单池搜索逻辑简单，实现成本低
- kd 搜索覆盖面广，每个类别数据量充足
- 模型训练效果好比搜索策略复杂更重要
- 标注阶段不需要处理多标签冲突的复杂度

### 1.7 为什么不以 tagCodeList 筛选

tagCodeList 是平台预设的分类标签，存在以下问题：
- 会漏掉小众岗位（如爬虫工程师、嵌入式开发）
- 有些岗位标题明确是某类岗位，但平台标签未归类
- kd 关键词搜索更灵活，覆盖面更广

## 二、薪资解析规则

### 输入

拉勾网薪资字段常见格式：
- "15k-30k"
- "20k-40k·14薪"
- "面议"
- "8k-12k"
- "10k"

### 解析规则

1. **范围提取**："Xk-Yk" 格式 → min=X, max=Y
2. **单一数值**："Xk" → min=X, max=X
3. **含薪资月数**："Xk-Yk·Nsalary" → 提取 N 作为 salary_months 字段，但不改变月薪数值
4. **面议**：min 和 max 均为空
5. **单位统一**：k 转换为千（15k = 15000）
6. **异常处理**：无法识别的格式记录日志，返回空值

### 输出

- salary_min（整数，单位：元/月）
- salary_max（整数，单位：元/月）
- salary_months（整数，如13薪、14薪，可为空）

## 三、城市标准化规则

拉勾网的 location 字段通常已经是标准城市名。

标准化规则：
1. 保留原始城市名
2. 预定义城市列表：北京、上海、广州、深圳、杭州、成都、武汉、西安
3. 不在列表中的标记为"其他"

## 四、数据去重规则

### 4.1 去重时机

在入库前统一去重，不在爬取时过滤。

原因：无法预知哪些岗位被多个关键词搜到，先全部爬下来再统一处理。

### 4.2 去重规则

- 以 job_url 为唯一键
- 同一 URL 只保留一条记录（保留字段最全的那条）
- 去重后统一进行标注

---

## 五、数据质量检查规则

### 5.1 入库前强制检查

1. **标题非空**：title 不能为空且长度 > 3
2. **链接有效**：job_url 不能为空且以 "http" 开头
3. **训练类别有效**：train_label 必须在预定义类别列表中
4. **薪资可为空**："面议"等情况允许薪资为空

### 5.2 入库后抽检

随机抽取 100 条已入库数据：
1. 标题非空率 > 99%
2. 最大类别占比 < 40%
3. 标题冲突修正的比例 < 10%（修正比例过高说明搜索关键词与标题偏差大）

## 六、滑块验证处理规则

### 检测

检查响应内容是否包含以下关键字：
- "验证"
- "验证码"
- "请完成安全验证"
- "slider"

### 处理

1. 检测到验证页 → 保存页面截图
2. 用 DrissionPage 手动打开页面，过滑块获取新 Cookie
3. 更新 Cookie 文件，继续爬取
4. 频繁请求下大概率不会触发，不需要自动破解逻辑
