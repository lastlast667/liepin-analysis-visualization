# Lagou Spider 项目总说明书

> 爬取目标：**拉勾网（www.lagou.com）** 招聘数据，作为招聘数据分析与可视化系统的数据源。

---

## 一、项目目标

### 1.1 爬取范围

- **目标网站**：拉勾网（www.lagou.com）
- **爬取页面**：职位搜索列表页 + 职位详情页
- **爬取方式**：通过 **kd 关键词**搜索，逐页爬取列表页，从列表页提取详情页链接后逐个爬取详情页
- **每页数量**：15 条
- **每关键词页数**：10 页（每关键词 150 条）
- **目标总量**：技术岗有效数据 2000+ 条

### 1.2 URL 结构

列表页：
```
https://www.lagou.com/wn/jobs?kd={{keyword}}&cl=false&fromSearch=true&labelWords=sug&pn={{page}}
```

参数说明：
- `kd`：搜索关键词（如"Java"、"Python"、"爬虫工程师"）
- `pn`：页码（1-10）
- `cl=false&fromSearch=true&labelWords=sug`：固定参数

**为什么用 kd 而不是 tagCodeList：**
- tagCodeList 是平台预设的分类标签，会漏掉小众岗位（如爬虫工程师、嵌入式开发）
- kd 关键词搜索更灵活，可以直接搜岗位标题中的关键词，覆盖面更广
- 不拼接 tagCodeList 和 labelWords，数据源更干净

详情页：从列表页提取的链接。

### 1.3 三层爬取架构

```
Layer 1: 关键词循环
    └── 遍历预定义的关键词列表（如 "Java", "Python", "爬虫工程师"...）
            
            Layer 2: 页码循环（每关键词10页）
                └── 请求列表页 URL
                    └── 从列表页提取每条职位的：详情页链接、摘要信息
                    
                    Layer 3: 详情页循环（每页15条）
                        └── 逐个请求详情页 URL
                        └── 提取完整字段：岗位标题、岗位标签列表、职位描述、薪资、地点等
                        └── delay 3-6秒
                        
                    delay 2-5秒
            
            delay 10-30秒
```

**设计说明**：
- 列表页只提取摘要 + 详情页链接
- 完整字段（特别是岗位标签和 JD 正文）从详情页提取
- 三层串行 + 随机延迟

### 1.4 核心字段

**从详情页提取**（标注的关键来源）：
- `title`：岗位标题
- `labels`：岗位标签列表（如 ["Java", "Spring", "微服务"]）
- `job_description`：职位描述（JD正文，ML向量化核心输入）

**辅助字段**：
- `company`、`salary`、`location`、`experience`、`education`、`job_url`、`publish_date`

## 二、技术选型

- **列表页**：requests（静态HTML）
- **详情页**：requests（拉勾网详情页服务端渲染）
- **解析**：lxml / BeautifulSoup
- **数据库**：MySQL
- **调度**：串行三层循环 + 随机延迟

**关于滑块验证：**
- 拉勾网滑块验证主要在长时间无操作后触发
- requests 模式下频繁请求大概率不会触发
- 如果触发（Cookie过期返回验证页）：用 DrissionPage 手动过滑块获取新 Cookie，然后继续
- 开发阶段先用 requests 尝试，不需要提前引入 DrissionPage

## 三、项目结构

```
spiders/
├── config.py              # 配置（关键词列表、请求头、延迟、数据库连接）
├── utils.py               # 工具（重试请求、日志、编码处理）
├── list_crawler.py        # 列表页爬虫：请求列表页 → 提取摘要 + 详情页链接
├── detail_crawler.py      # 详情页爬虫：请求详情页 → 提取完整字段
├── keyword_mapper.py      # 搜索关键词 → 训练类别映射（用于默认标注）
├── labeler.py             # 数据标注：基于搜索关键词默认标注 + 标题校验修正
├── data_pipeline.py       # 数据清洗 + 去重 + 入库
├── models.py              # 数据库表结构
├── keyword_scheduler.py   # 关键词队列管理 + 断点续爬
├── jobs/                  # 运行时数据（gitignore）
│   ├── raw/               # 原始爬取JSON
│   ├── logs/              # 日志
│   └── progress.json      # 进度快照
└── tests/
```

## 四、反爬策略

- 列表页请求间隔：2-5秒随机
- 详情页请求间隔：3-6秒随机
- 关键词切换延迟：每完成一个关键词，暂停 10-30秒
- User-Agent轮换、Cookie复用
- 连续失败3次切换Cookie，连续失败5次暂停5分钟
- 开发阶段不需要代理
- 滑块验证：检测到验证页时，保存截图，用 DrissionPage 手动处理（开发阶段），或用新Cookie重试

## 五、数据标注策略（核心）

### 5.1 核心原则

**标注依据：搜索关键词默认标注 + 标题校验修正。**

- 搜索关键词决定**默认标签**（如搜"Java"→默认"Java后端开发"）
- 标题校验决定**最终标签**（如搜"Java"但标题是"Python开发（了解Java）"→修正为"Python/Go后端开发"）
- 岗位标签列表（labels）**不参与标注**，仅作为特征用于后续ML向量化

### 5.2 为什么搜索关键词可以作为默认标注？

与 tagCodeList 不同，kd 关键词搜索的结果与关键词高度相关：
- 搜"Java" → 结果几乎都是 Java 相关岗位
- 搜"爬虫工程师" → 结果几乎都是爬虫岗位
- 搜"嵌入式" → 结果几乎都是嵌入式岗位

这是因为 kd 搜索匹配的是**标题和描述中的关键词**，而不是平台预设的分类标签。

### 5.3 标注流程

```
全部原始数据 → URL去重 → 统一标注 → 清洗 → 预处理
```

### 5.4 标注规则

**第一步：搜索关键词默认标注**

每个搜索关键词对应一个默认训练类别：

| 搜索关键词 | 默认 train_label |
|-----------|-----------------|
| Java | Java后端开发 |
| Python | Python/Go后端开发 |
| Go | Python/Go后端开发 |
| C++ | C++后端开发 |
| PHP | 其他后端开发 |
| 爬虫 | Python/Go后端开发 |
| 嵌入式 | C++后端开发 |
| 前端 | 前端开发 |
| 数据分析 | 数据分析 |
| 算法 | 算法工程师 |
| 测试 | 测试/运维开发 |
| 运维 | 测试/运维开发 |
| 全栈 | 其他后端开发 |

**第二步：标题校验（冲突修正）**

如果岗位标题与搜索关键词**严重不符**，以标题为准修正标签。

冲突判定：
- 搜"Java"，标题含 "Python" 且不含 "Java" → 修正为 Python/Go后端开发
- 搜"Python"，标题含 "Java" 且不含 "Python" → 修正为 Java后端开发
- 搜"前端"，标题含 "Java后端" → 修正为 Java后端开发

不冲突（保留默认标签）：
- 标题同时含 "Java" 和 "Go"（正常多技能）
- 标题模糊如 "后端开发工程师"（无法明确判断，保留默认）
- 标题含搜索关键词的同义词（如搜"Java"，标题含"JVM"）

**第三步：无冲突保留默认**

标题和搜索关键词没有明显冲突 → 保留搜索关键词决定的默认标签。

### 5.5 为什么 labels 不参与标注

- 标签由招聘方填写，可能存在疏忽（如 Java 岗标签未选 Java）
- 标题是招聘方最关注的表达，优先级最高
- 搜索关键词已经提供了强烈的类别信号
- labels 作为特征用于后续 ML 向量化，但不用于决定类别

### 5.6 单池搜索设计

**本系统采用单池搜索：** 模型预测主类别后，只在该类别的 JD 池内做相似度匹配。

**不做多池搜索的原因：**
- 单池搜索逻辑简单，实现成本低
- 数据做干净后（kd 搜索覆盖面广），每个类别的数据量充足
- 模型训练效果好比搜索策略复杂更重要
- 后续如需扩展，可在标注阶段增加 sub_labels 字段支持多池

## 六、关键词选择（V1.0）

### 6.1 关键词列表

| 搜索关键词 | train_label | 选择理由 |
|-----------|-------------|---------|
| Java | Java后端开发 | 大众标签，数据量充足 |
| Python | Python/Go后端开发 | 大众标签 |
| Go | Python/Go后端开发 | 与Python合并（样本可能少） |
| C++ | C++后端开发 | 主流后端语言 |
| PHP | 其他后端开发 | 小众但精准 |
| 爬虫 | Python/Go后端开发 | 爬虫主要用Python，归入Python池 |
| 嵌入式 | C++后端开发 | 嵌入式主要用C/C++，归入C++池 |
| 前端 | 前端开发 | 明确方向 |
| 数据分析 | 数据分析 | 明确方向 |
| 算法 | 算法工程师 | 明确方向 |
| 测试 | 测试/运维开发 | 明确方向 |
| 运维 | 测试/运维开发 | 与测试合并 |
| 全栈 | 其他后端开发 | 跨方向，兜底 |

### 6.2 为什么选择这些小众关键词？

- **爬虫工程师**：tagCodeList 筛选漏掉的岗位，kd 搜索能精准覆盖
- **嵌入式开发工程师**：tagCodeList 筛选数据极少（4页），kd 搜索有几十页
- **PHP**：小众但明确，归入"其他后端开发"保证数据不浪费

### 6.3 不选的关键词

- 框架标签：Spring、Django、Vue、React（作为ML特征，不作为搜索入口）
- 粗粒度标签：后端开发（内部包含多种语言，增加分类压力）

## 七、关键约束

1. 所有 HTTP 请求走 utils.request_with_retry()
2. 所有配置从 config.py 读取
3. 所有数据库操作走 data_pipeline.py
4. 标注依据：搜索关键词默认 + 标题校验修正，labels 不参与标注
5. 单池搜索设计，不存储 sub_labels
6. 日志记录：请求URL、状态码、耗时、异常信息
