# Lagou Spider 项目总说明书

> 爬取目标：**猎聘** 招聘数据，作为招聘数据分析与可视化系统的数据源。

---

## 一、项目目标

### 1.1 爬取范围

- **目标网站**：猎聘
- **爬取页面**：职位搜索列表页 + 职位详情页
- **爬取方式**：通过 **key 关键词**搜索，逐页爬取列表页，从列表页提取详情页链接后逐个爬取详情页
- **每页数量**：15 条
- **每关键词页数**：10 页（每关键词 150 条）
- **目标总量**：技术岗有效数据 2000+ 条

### 1.2 URL 结构

列表页：
```
https://www.liepin.com/zhaopin/?city=410&dq=410&pubTime=&currentPage={{page}}&pageSize=40&key={{keyword}}
```

参数说明：
- `key`：搜索关键词（如"Java"、"Python"、"爬虫工程师"）
- `currentPage`：页码（1-10）

详情页：从列表页提取的链接。

### 1.3 三层爬取架构

```
Layer 1: 关键词循环
    └── 遍历预定义的关键词列表（如 "Java", "Python", "爬虫工程师"...）
            
            Layer 2: 页码循环（每关键词10页）
                └── 请求列表页 URL
                    └── 从列表页提取每条职位的：详情页链接、摘要信息
                    
                    Layer 3: 详情页循环（每页15条）
                        └── 逐个请求详情页 URL
                        └── 提取完整字段：岗位标题、岗位标签列表、职位描述、薪资、地点等
                        └── delay 3-6秒
                        
                    delay 2-5秒
            
            delay 10-30秒
```

**设计说明**：
- 列表页只提取摘要 + 详情页链接
- 完整字段（特别是岗位标签和 JD 正文）从详情页提取
- 三层串行 + 随机延迟

### 1.4 核心字段

**从详情页提取**（标注的关键来源）：
- `title`：岗位标题
- `labels`：岗位标签列表（如 ["Java", "Spring", "微服务"]）
- `job_description`：职位描述（JD正文，ML向量化核心输入）

**辅助字段**：
- `company`、`salary`、`location`、`experience`、`education`、`job_url`、`publish_date`

## 二、技术选型

- **列表页**：requests（静态HTML）
- **详情页**：requests（猎聘详情页服务端渲染）
- **解析**：lxml / BeautifulSoup
- **数据库**：MySQL
- **调度**：串行三层循环 + 随机延迟

## 三、项目结构

```
spiders/
├── config.py              # 配置（关键词列表、请求头、延迟、数据库连接）

│   ├── logs/              # 日志
│   └── progress.json      # 进度快照
```

## 四、反爬策略

- 列表页请求间隔：2-5秒随机
- 详情页请求间隔：3-6秒随机
- 关键词切换延迟：每完成一个关键词，暂停 10-30秒
- User-Agent轮换、Cookie复用
- 连续失败3次切换Cookie，连续失败5次暂停5分钟
- 开发阶段不需要代理
- 滑块验证：检测到验证页时，保存截图，用 DrissionPage 手动处理（开发阶段），或用新Cookie重试

## 五、数据标注策略（核心）

### 5.1 核心原则

**标注依据：搜索关键词默认标注 + 标题校验修正。**

- 搜索关键词决定**默认标签**（如搜"Java"→默认"Java后端开发"）
- 标题校验决定**最终标签**（如搜"Java"但标题是"Python开发（了解Java）"→修正为"Python后端开发"）

### 5.2 为什么搜索关键词可以作为默认标注？

- 搜"Java" → 结果几乎都是 Java 相关岗位
- 搜"爬虫工程师" → 结果几乎都是爬虫岗位
- 搜"嵌入式" → 结果几乎都是嵌入式岗位

这是因为 key 搜索匹配的是**标题和描述中的关键词**

### 5.3 标注流程

```
全部原始数据 → URL去重 → 统一标注 → 清洗 → 预处理
```

### 5.4 标注规则

**第一步：搜索关键词默认标注**

每个搜索关键词对应一个默认训练类别：

| 搜索关键词  | 默认 train_label |
|--------|----------------|
| Java   | Java后端开发       |
| Python | Python后端开发     |
| Go     | Go后端开发         |
| C++    | C++后端开发        |
| PHP    | PHP后端开发        |
| 爬虫     | 爬虫工程师          |
| 嵌入式    | 嵌入式开发          |
| 前端     | 前端开发           |
| 数据分析   | 数据分析师          |
| 大数据    | 大数据开发工程师       |
| 算法     | 算法工程师          |
| 软件测试   | 测试工程师          |
| 运维     | 运维工程师          |
| 全栈     | 全栈工程师          |
**第二步：标题冲突校验**

如果岗位标题与搜索关键词**严重不符**，以标题为准修正标签。

**冲突判定（需修正）：**
- 搜索关键词=Java，标题含 "Python" 且不含 "Java" → 修正为 Python后端开发
- 搜索关键词=Python，标题含 "Java" 且不含 "Python" → 修正为 Java后端开发
- 搜索关键词=Python，标题不含任意一个关键词 → 过滤
- 搜索关键词=算法，标题含 "测试工程师" → 修正为 测试工程师

**不冲突（保留默认标签即搜索关键词）：**
- 标题同时含 "Java" 和 "Go"（正常多技能要求）
- 标题模糊如 "后端开发工程师"（无法明确判断，保留默认）
- 标题含搜索关键词的同义词（如搜"Java"，标题含"JVM"）

**第三步：保留默认**

标题与搜索关键词没有明显冲突 → 保留搜索关键词决定的默认标签。

### 5.5 单池搜索设计

**本系统采用单池搜索：** 模型预测主类别后，只在该类别的 JD 池内做相似度匹配。

**不做多池搜索的原因：**
- 单池搜索逻辑简单，实现成本低
- 数据做干净后（kd 搜索覆盖面广），每个类别的数据量充足
- 模型训练效果好比搜索策略复杂更重要
- 后续如需扩展，可在标注阶段增加 sub_labels 字段支持多池

## 七、关键约束

1. 所有 HTTP 请求走 utils.request_with_retry()
2. 所有配置从 config.py 读取
3. 所有数据库操作走 data_pipeline.py
4. 标注依据：搜索关键词默认 + 标题校验修正，labels 不参与标注
5. 单池搜索设计，不存储 sub_labels
6. 日志记录：请求URL、状态码、耗时、异常信息
